Decisions made:

  • Story-aware agents over generic frontend/backend/QA. A "backend-agent" makes no sense for a browser-only localStorage app — there is no backend. The split is by actual work lane: visual, logic, story content. Perplexity initially suggested generic roles; Claude pushed back and designed project-native agents instead.
  • No dedicated QA agent — QA built into orchestrator as a post-task review pass. The project has no formal QA infrastructure beyond story-utils.js tests. Rather than invent a fake QA agent, a verification checklist was added to the orchestrator's post-task behavior: event handler audit, localStorage key audit, browser-test flag.
  • story-agent is on-demand only, not a standing team member. Most sessions are code sessions; the Lore Keeper wakes up for wiki/canon tasks and goes quiet otherwise. This keeps the orchestrator from over-invoking it on every feature request.
  • Agent prompts reference existing rule files rather than duplicating content. Each agent says "read .claude/rules/css.md" rather than inlining the rules. This keeps agent prompts lean and ensures they stay current as rules evolve.
  • Persona blocks sit above a --- divider in each agent file. This makes it visually clear where flavor ends and technical instructions begin — both for Claude reading the file and for future human editing.
  • description frontmatter contains only task-routing language — no persona text. The description field drives auto-routing logic; mixing persona flavor into it would pollute the signal used for delegation decisions.
