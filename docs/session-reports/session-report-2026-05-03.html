07Open Questions & Gaps

  • Browser test debt is now eight sprints deep. Sprint Q toolbar, Sprint Q Sage tools, Sprint R tools + dialog, voice mic fixes, Design Polish, Orb Style Selector, Sprint O, Sprint N, and Issues 1–3 fixes are all unverified in Live Server. None of this session's work has been browser-tested.
  • Sage's switch_project calls location.reload() mid-tool-loop. If Claude is mid-streaming or awaiting tool results when the reload fires, state will be lost cleanly (localStorage is already saved before reload). This is expected behavior — document it so it doesn't cause confusion.
  • format_text uses regex on innerHTML. If the target text spans multiple HTML elements (e.g., bold mid-phrase), the regex won't match. This is a known limitation — Sage should be told to use plain text targets only.
  • Font family via execCommand('fontName') wraps text in <font face="..."> tags — a deprecated HTML element. Most browsers still support it, but it may cause issues if the editor content is exported or processed. No fallback yet.
  • Sage's request_user_choice dialog is not dismissed on page navigation. If the user navigates tabs while a choice dialog is visible, the dialog persists. Low-impact but worth noting.
  • [hidden] { display: none !important; } still missing from style.css. Seven sessions deferred. This is now a standing hard blocker — must go in before the next sprint.
  • UI declutter pass not started. This is a full-site audit — every tab needs review. Cannot be done without opening the live app and walking through each tab manually.

Decisions made:

  • Font family and line spacing added despite being "nice-to-have." User's position: differentiators matter. The 7 fonts selected (Georgia, Times New Roman, Garamond, Palatino, Arial, Calibri, Courier New, Verdana) cover the main writer use cases. More fonts deferred to a future session when the user has specific fonts in mind.
  • Highlight replaced single-yellow with 6-color palette. The old single yellow button and remove-highlight button were replaced entirely by a picker popover. Fewer toolbar items, more capability.
  • Font color "A" button uses a CSS variable for the underline indicator. --font-color-indicator is set on the button via JS when a color is chosen, so the indicator always shows the last-used color without additional state management.
  • request_user_choice renders inline in chat, not as a modal overlay. An overlay would interrupt the writing flow. Inline choice cards match the chat-native UX and are dismissible by clicking a choice (which sends the message and removes the card).
  • switch_project requires confirmed_by_user: true at the tool level AND Sage must call request_user_choice first. Double guard: the tool blocks without the flag, and the system prompt instructs Sage to always show the dialog first. Belt and suspenders.
  • format_text operates on innerHTML with regex. Known limitation: won't match text that spans HTML element boundaries (e.g., partially bold phrases). Sage should be instructed to target plain text runs. No workaround added yet — the simpler approach covers 95% of use cases.
  • Sage's arc tools map to card types arc/event/scene. The Arcs & Timeline tab is card-based — there's no separate timeline event data structure. Creating an "arc entry" just creates a card with the right type and appends its ID to arcOrder.

What worked well:

  • Pre-read before delegating. For both Sprint Q and Sprint R, reading the relevant code sections (toolbar HTML, JARVIS_TOOLS pattern, localStorage schemas, save functions) before writing the agent prompt meant the agents had exact function names, data shapes, and insertion points. Zero "I couldn't find X" failures.
  • Discovering existing features before building. The pre-read for Sprint Q revealed that text alignment and Find & Replace already existed — narrowing the actual work from 6 items to 4 new + 2 wires. Saved significant implementation time.
  • Full tool inventory as a standalone step. Pulling the complete JARVIS_TOOLS list and mapping it against app capabilities in a table gave a clear, shared view of the gap — made the Sprint R scope obvious and easy to approve.
  • Schema research before the Sprint R prompt. Checking how character_profiles, arc_order, saveCards(), renderCards(), and location.reload() actually work meant the agent prompt contained exact implementation details — the agent executed cleanly on the first pass with 54 tool uses.
  • Inline confirmation dialog as a chat-native affordance. Rejecting the modal overlay in favor of inline choice cards was the right call — it keeps the interaction in the conversation thread and feels natural for a chat-first tool.

What to do differently:

  • Browser test before /insights — eight sessions. This note has appeared every session. Sprint Q and R are both unverified. The pattern is clear: features get built faster than they get tested. The next session should start with browser tests before any new code.
  • Commit incrementally. Sprint Q was done, then Sprint R was done, then /insights, then commit. Three sprints of work accumulated before a commit. If Sprint R had a file-corruption issue, Sprint Q would have been at risk too. Commit after each confirmed sprint.
  • The format_text innerHTML regex limitation should have been called out to the user before shipping. It was documented in the report but not surfaced during the session. The user should know that asking Sage to color text that's partially bold won't work — they may hit this frustratingly in real use.
