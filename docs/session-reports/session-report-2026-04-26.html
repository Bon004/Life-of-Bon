Decisions made:

  • Namespace all per-project keys as <projectId>_suffix (no sf_ in suffix). The export data object uses bare suffixes (e.g., "cards") to avoid double-prefix: proj_xxx_sf_cards is a bug, proj_xxx_cards is correct. Perplexity flagged this risk explicitly in their plan review — it was the primary source of discussion.
  • Auto-migrate on first load, leave old keys as safety net. Old sf_cards etc. are left in localStorage (not deleted) so the user can recover if migration breaks something. A future cleanup step can remove them once the user confirms everything works.
  • location.reload() on project switch. Rather than manually resetting 25+ in-memory state variables, a full page reload is cleaner and guarantees all state re-initializes correctly from the new namespaced keys. Performance cost is negligible for this use case.
  • Two-step string replacement for "Life of Bon" in AI prompts. The occurrences were inside single-quoted JS strings. First pass replaced "Life of Bon" with " + currentProjectName + " (wrong — produced a literal string). Second pass fixed it to proper JS concatenation: ' + currentProjectName + '. This was a predictable error that should have been accounted for in the first pass.
  • Export/import section rewritten before the 25 key replace-all passes. The old BACKUP_KEYS array contained 'sf_cards', 'sf_positions' etc. — if the replace-all passes had run first, those would have become projectKey('cards') inside the array literal, producing invalid JavaScript syntax.
  • Duplicate project name guard is case-insensitive. "life of bon" and "Life of Bon" are treated as the same name to prevent confusing near-duplicates.
  • Perplexity collaboration drove architecture. Two rounds of plan review before implementation. Perplexity endorsed the approach and added the double-prefix import bug and v1 backward-compat test as the two key risks to address.

What worked well:

  • Three-way collaboration again (user + Perplexity + Claude). The plan went through two full Perplexity reviews before any code was written. Perplexity caught the double-prefix import bug (proj_xxx_sf_cards vs. proj_xxx_cards) and the v1 legacy import test case — both of which made it into the final implementation.
  • Rewriting export/import before the replace-all passes. Doing the export section first prevented the BACKUP_KEYS array from being mangled by the mechanical replacement. Order of operations mattered here and was caught before it caused a problem.
  • Replace-all ordering to avoid substring conflicts. Replacing 'sf_dismissed_batches' before 'sf_batches', 'sf_summary_expanded' before 'sf_summary', 'sf_arc_sequence_map' before 'sf_arc_order' — longer/overlapping keys first prevents partial replacement bugs.
  • IIFE for bootstrap. Using an IIFE to run the project bootstrap prevented 10+ intermediate variables from polluting the global module scope, while still exporting the 4 needed values via window._sf_*.

What to do differently:

  • Account for string quoting context in replace-all. The "Life of Bon" replacement produced " + currentProjectName + " (double-quoted interpolation inside single-quoted strings) on the first pass — wrong. The correct output required ' + currentProjectName + '. The fix was a second replace-all pass, but the error should have been predicted before the first pass. Always check whether target strings are inside single or double quotes before writing the replacement.
  • Browser test before /insights. Same note as last session — multi-project is implemented but unverified in-browser. The 12-step checklist is the first thing that should happen next session.
