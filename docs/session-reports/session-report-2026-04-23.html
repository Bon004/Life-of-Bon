Decisions made:

  • Root cause: CHUNK_SIZE / max_tokens mismatch, not deduplication or prompt logic. A 30,000-char chunk of the story bible required 40–55 cards in response. At ~350 tokens/card average, that's 14,000–19,000 tokens — far above the 8,192 cap. Truncated JSON fails JSON.parse, catches silently, returns []. Chunks 1–3 returned nothing; only the lighter section 11–13 chunk (quotes + open decisions) fit within the limit and returned 2 cards.
  • CHUNK_SIZE set to 10,000 (not 8,000 or 5,000). At 10K chars, a chunk produces ~8–15 cards × ~300 tokens = 2,400–4,500 tokens — comfortably under 8,192. Smaller values would create more API calls with no added benefit. Rejected: 5,000 (too many calls), 15,000 (still risky for dense sections).
  • Partial recovery added as a fallback, not a primary fix. The chunk size reduction is the real fix — partial recovery is insurance for edge cases where a single chunk is unusually dense. It salvages complete card objects before any truncation point by slicing at the last },. Rejected: more aggressive JSON repair (too complex, not needed if chunk size is correct).
  • Sections brevity hint added to prompt rather than removing the field. The sections field is valuable for structured character/world cards. Constraining sub-field values to 1–2 sentences is lighter-weight than removing the field entirely and preserves the C2 feature. Rejected: dropping sections from the extraction pass.

10Claude Collaboration Insights

What I asked Claude to do:

  • Debug the import pipeline: user uploaded a ~100K character story document and received only 2 cards. Identify exactly where content was being lost.
  • Propose a plan (plan mode) before touching code — trace through all possible failure points: chunking, prompt, extraction loop, deduplication, JSON parsing, save.
  • Implement the fix, run /insights, and commit everything to GitHub.

What worked well:

  • Plan mode with a single focused Explore agent. The agent mapped the entire pipeline in one pass — 10+ functions, all line numbers, all constants — in a single tool call. No follow-up reads needed. The plan was written directly from that output with high confidence.
  • Reading the actual code before proposing the fix. After the Explore agent returned, the critical sections (callClaudeForCards lines 1389–1407, organizeWithAI lines 1449–1474, splitIntoSections lines 1339–1363) were read directly to verify the hypothesis before the plan was written. The math confirmed it: 40 cards × 350 tokens = 14,000 tokens > 8,192.
  • Surgical edits — 4 changes, all in one file. The fix didn't require restructuring the pipeline or adding new abstractions. Each change addressed one specific failure mode: chunk size (primary), truncation visibility (debug), partial recovery (fallback), section brevity (secondary token reduction). All four changes are independent and any could be reverted without affecting the others.
  • Deduplication was correctly ruled out early. The seenTitles Set logic was inspected and confirmed correct — it accumulates across chunks, not overwrites. This ruled out false-positive deduplication as a cause. Without this verification, the investigation might have chased the wrong bug.

What to do differently:

  • The fix has not been re-tested in a live browser. The analysis and code changes are sound, but the proof is in the actual import output. The next session must open the browser and run the import before anything else.
  • A console.log of chunk count and char length at the start of the loop would have surfaced the problem immediately. If the loop had logged "Chunk 1 of 4: 28,432 chars", the overflow would have been obvious before any API call. Adding debug logging to the import pipeline at the start of each iteration would make future issues like this self-diagnosing.

Patterns worth repeating:

  • When a pipeline returns unexpectedly few results with no error shown, look for silent failures in catch blocks — return [] on parse error is the pattern that hid this bug for the entire session.
  • Always check stop_reason from any API that supports it — truncation is a common silent failure mode for AI APIs, and none of the existing code was checking for it.
  • For token budget analysis: count expected output cards × average tokens/card and compare to max_tokens. This math catches the CHUNK_SIZE / max_tokens mismatch before any API call is made.
Rule: always check stop_reason on Claude API responses. The API signals truncation via data.stop_reason === 'max_tokens'. Without this check, truncated responses silently fail JSON parsing and return empty arrays — zero cards, zero error messages, no indication anything went wrong. Now logged as a console warning.
Why: This is exactly what happened — 3 of 4 chunks returned nothing, silently, for an entire session.
Rule: size CHUNK_SIZE relative to max_tokens, not to "seems reasonable". 30,000 chars for a dense story bible = 40–55 cards = 14,000–19,000 tokens. The 8,192 cap was never going to hold. The correct sizing formula: (max_tokens − prompt_tokens) ÷ avg_tokens_per_card × avg_chars_per_card. For this pipeline: (8192 − 700) ÷ 350 × 70 ≈ 1,500 chars/card × 10 cards target = 15,000 chars max. 10,000 is the conservative, safe choice.
Why: The original 30,000 was chosen without considering output token budget — a reasonable-seeming number that happened to be 3× too large.
