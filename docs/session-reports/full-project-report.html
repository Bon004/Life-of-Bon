What This Project Is

StoryForge is a personal web app for organising and writing an isekai/anime story called Life of Bon. The main character gets reincarnated. The app lets you capture story notes, auto-categorise them into cards using Claude AI, view them on a Story Canvas, and write and plan using dedicated workspace tabs.

Architecture Decisions

DecisionWhy
No frameworksUser wants simple, maintainable code. No npm, no build step. Open with Live Server and go.
Single files (index.html, style.css, app.js)CLAUDE.md rule: don't add new files. Everything in 3 files.
localStorage onlyNo server, no accounts. Personal use only. Data is local.
Claude API in browserConvenient for personal use. Uses anthropic-dangerous-direct-browser-access header. Not suitable for public deployment.
story-utils.js exceptionPure utility functions go in story-utils.js for unit testability (vitest). DOM code stays in app.js.
Static data as JS constantsEnneagram types, 36 Situations, 8 Sequences — embedded in app.js rather than fetched, for simplicity.
document.execCommand() for rich textDeprecated but fully functional; Selection API would add 200+ lines for no user-visible gain in a plain-JS personal app.

Workflow Rules

Read files before editing — never propose changes to code that hasn't been read
Plan first for major features — user wants to review a roadmap before implementation starts
No commits without explicit user request — never push to GitHub autonomously
Browser testing after DOM changes — manual only, no automated DOM tests
TDD for pure functions — write test first in story-utils.test.js, then implement in story-utils.js
Token optimisation — caps on sf_chat_memory (3000 chars), paste input, Related action, chatHistory
No overwriting user content with AI — AI fills empty fields only (Characters tab Generate Profile)
Trunk-based git — main branch should always be stable; feature branches for experiments
/insights generates two outputs — a session-specific report + updated full-project report (this file)

Open Questions & Unresolved Decisions

Should the Writing tab support multiple chapters/scenes or stay single-file? Not decided — current implementation is one flat rich-text pane per Working Copy.
Should export/import (Phase 4b) replace or merge cards on import? Not decided — needs UI design for the confirmation/choice dialog.
iPad layout breakpoints — what screen width triggers narrow layout? Not defined yet — Phase 4c work will establish this.
Character root-name grouping refinement Currently uses first token. Unrelated characters with matching first tokens (e.g., "Bon" + "Bonwick") would be incorrectly grouped. May need explicit card-linking UI in future.
