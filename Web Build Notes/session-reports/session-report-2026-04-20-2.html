Run 3 — Card Type Taxonomy

What I asked Claude to do:

  • Extend the card type system from 5 to 12 types with a TYPE_TO_COLUMN routing map.
  • Overhaul the extraction prompts: remove the card cap, switch to section-by-section scanning, add source_section and tags output fields.
  • Add CSS for all new types (inheriting column palette) plus tag chip and source label styles.

What worked well:

  • The column-routing approach (12 types → 5 columns) kept the board layout intact while unlocking fine-grained type labeling — a clean architectural split.
  • Reading both app.js and style.css diffs in one pass made it easy to confirm visual and data-layer changes were consistent.

What needed correction:

  • The app.js diff was truncated — tag rendering in card HTML was not fully confirmed. Should have explicitly checked the render loop before declaring the feature done.

Patterns worth repeating:

  • When expanding a type system, define the routing/mapping table first (TYPE_TO_COLUMN), then update data layer, then CSS — the order prevents half-wired states.
  • New skill files belong under .claude/skills/<name>/ with SKILL.md + referenced templates in the same folder — never spread workflow prose across CLAUDE.md and Web Build Notes/.

Candidates to persist:

Rule: When adding a new Claude workflow, default to a skill under .claude/skills/<name>/ with SKILL.md + referenced templates in the same folder.
Why: The /insights split across CLAUDE.md + insights-format.md caused stale section counts and duplicated save paths over multiple sessions.
Rule: After any extraction prompt change, do a live browser import to verify the new fields (source_section, tags) actually appear in cards — don't rely on the diff alone.
