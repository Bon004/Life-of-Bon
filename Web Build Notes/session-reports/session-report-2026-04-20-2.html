What worked well:

  • Planning mode kept the restructure from turning into an uncontrolled edit — the plan file captured decisions before any file moved.
  • Using AskUserQuestion for three specific trade-offs (scope, delete vs. stub, single-source save paths) prevented guessing and locked in the design.
  • Mirroring the existing .claude/skills/status/ shape meant the new skill felt native, not bolted on.

What needed correction:

  • None this session — the user's own recommendation ("CLAUDE.md central, /insights as skill, long templates referenced") was already the right call and was followed directly.

Patterns worth repeating:

  • When instructions are split across files, look for duplication of the same fact (save paths, section counts) — those are the strongest signals that the split is wrong.
  • Keep skills self-contained in their own folder: SKILL.md + referenced template.md beats a skill that cross-links to distant docs.
  • When a template has to stay identical across runs, store it as a separate file the skill reads — not inlined in the workflow prose.

Candidates to persist (into CLAUDE.md or memory):

Rule: When adding a new Claude workflow, default to a skill under .claude/skills/<name>/ with SKILL.md + any referenced templates in the same folder — do not spread workflow prose across CLAUDE.md and Web Build Notes/.
Why: this session fixed exactly that drift for /insights. The split had caused stale section counts, duplicated save paths, and two-file cross-reads on every run.
Consider saving the above as a feedback memory so future sessions don't re-introduce workflow prose into CLAUDE.md.
